# 代码翻译执行流验证门 — 计划

---

## A. 背景与目的（这道门是干嘛的）

- **项目在做的事**：把 `gamemd.exe` 逆向重建成 C++。子Agent 拿 IDA 反编译出的 **C 伪代码**，把当前还没实现的函数**转写成高质量 C++ 实现**。翻译要"**神似**"（还原算法/对外行为），子Agent 本就以反编译 C 代码为蓝本、**应照伪代码的控制流形态转写**（把 if 链改写成 switch 反而不合要求）。
- **要从根上杜绝的痛点**：必须假定子Agent 随工作推进会忘掉几乎所有要求；它在被"指针加法门""非数组 `[]` 门"逼住后，**会自发涌现出一种逃逸行为——把对外部的操作（调用、写成员/写全局）改成注释，留半截 = 局部 stub**。靠它自觉不可靠；而**全部约 19,067 个函数都要翻译，逐个人工 review 不现实**——所以必须自动化拦截。
- **方案**：在 CMake 构建里硬性拦截。本门是其中**专抓"局部 stub / 宏观结构缺失"的一道**。

### 这道门的精确身份（务必照此理解，别拔高）
本门把每个函数实现当**黑箱（黑口袋）**，并且是**必要条件、非充分条件**的检查：
- **PASS 不代表"翻译正确/等价"**，只代表"从外面摸不出宏观差别"。
- 它**不判断翻译对不对**，只用"后继执行流（黑口袋里有几个硬块、怎么连）+ 每块对外操作"做**粗估**，专抓"明显少做了一截"。
- **唯一需要严格确认的东西 = 这个黑箱对外部函数/变量的操作**（调用了谁、写了哪些成员/全局），**因为子Agent 的逃逸几乎总是发生在这里**（把 CALL/WRITE 注释掉）。

### 作者契约（"神似"的可操作定义；这是 switch↔if 类"形不同"的处理方式）
本门下"忠实翻译"= **镜像伪代码的控制流形态与对外操作**：
- **照伪代码的分支形态写**：伪代码是 if 链就写 if 链、是 switch 就写 switch；**不要把 if↔switch 互改**，也不要改写循环形态。这样"稀疏 switch 被 Hex-Rays 降成 if 树、作者却写 switch"的形不同**根本不会出现**（把 if 翻成 switch 本就不合"神似"要求）。如将来确需放宽，可选补"单一变量相等比较链 ↔ 多路节点"双向归一化（非必需）。
- **调用照写成调用**：伪代码里**本就是调用**就写调用、不手工内联成成员访问；反过来，伪代码里**已被内联**的副作用函数，**照伪代码镜像其内联体**、不要 de-inline 成对未翻译函数的调用（见 D-STEP3：非平凡内联 de-inline 到未翻译被调会被挡）。多数平凡 getter，Hex-Rays 在伪代码侧本就内联成 `*(this+N)` 成员读，两侧自然一致；残留由本契约 + 现有作用域解析机制覆盖。
- 违反契约（重塑形态/手工内联）= 作者的不推荐写法、可能被挡；按本门对"忠实"的定义，这不算误报。
- **缓存一致性前提**：作者须照 `.omo/ida_cache/` 里的伪代码转写（或缓存与作者所见同一份 decompile/类型信息）——否则随翻译推进的类型/命名演进（B.4）可能让 Hex-Rays 的内联/形态在"缓存时 vs 作者时"漂移，使契约失效。

### 严格确认 vs 故意不管（本门的契约）
| | 内容 | 处理 |
|---|---|---|
| **严格（底线，必须对齐）** | ① CFG 结构（基本块 + 后继边的种类）② 绑定在每个块上的**对外操作**：CALL、WRITE(成员/全局) | 对不上 → FAIL |
| **宽松 / advisory（不参与裁决）** | READ（已降级为 advisory）、块内各操作的**先后次序** | 不参与配对、不致 FAIL（见 D-STEP3） |
| **故意不管（设计内 known-limitation，不修）** | 分支条件本身（运算符 `<`/`<=`、常量、操作数配对、`inline operator<` 之类内部运算）；对象身份（`this->X` vs `target->X` 同偏移不区分，因当前大量 `void*`/`uint32_t*` 真实类型未确认、逐个深挖成本极高）；变量下标成员访问（`this->Array[i]` 无静态偏移） | 一律视为**允许的漏报**（非宏观结构性） |

### 设计取舍：零误报优先，允许非宏观漏报
**首要工程目标 = 本门绝不误报（绝不误挡忠实代码）。** 编译器优化（`while`↔`do-while`↔`for` 循环互转、写重排、CSE 等）极其复杂，无法可靠还原，所以本门**只在"对外操作 + 宏观结构"层面做粗判**，放弃细粒度。**非宏观结构性的漏报是设计内允许的**；只要不把忠实代码挡下来即可。

### 为什么"一定错误率被允许"——本门不是最后一道防线
本门是**最便宜、最早**的一道静态网（构建时抓宏观局部 stub），**不负责确认行为正确**。行为正确性由**下游的批量对拍**兜底，分两个层面：
- **输出层面对拍**：用钩子（Syringe 注入 hook_dll）把我们的 RE 函数挂进原版 `gamemd.exe`，在同一状态上跑、比对返回值与对外效果，差异记入 `comparisonResult.log`（Same/Different Compares）。这能抓出本门**故意不管**的细粒度错误（②分支条件/运算符/常量、③对象身份、边界差异等）。
- **实际效果层面对拍（忒修斯之船）**：逐步用我们的函数**彻底取代**原程序的对应函数，让**运行中的游戏本身**成为测试——换上我们的实现后游戏行为仍正确，即为效果层面的终极验证。

正因为有这两层下游行为网，本门**容许一定错误率（漏报）**：它只要做到"零误报 + 抓住宏观局部 stub"，把细粒度正确性留给对拍即可。这也是 ②③ 等被列为设计内 known-limitation、而非缺陷的根本原因。

### 为什么二态 PASS/FAIL、无第三态
这是全自动构建门，构建只有过/不过。第三态"我不确定"在自动化里没人接手。因此：在**严格核心**（结构 + 对外操作）上能确信缺失 → FAIL；其余 → PASS 或 skip（守零误报，绝不 FAIL）。

## B. 已有的拦截门 + 各门具体算法 + CMake 流程（给 Oracle 看的全貌）

### B.0 CMake 接线（实测自 `CMakeLists.txt`）+ 目标架构
`gamemd` 依赖各门，任一非零退出 → 构建失败：
```
gamemd
 ├─ check_code_quality      → check_translated_functions.py            （现役，依赖 build_signals）
 ├─ verify_execution_flow   → verify_execution_flow.py --auto           （现役，依赖 build_signals）
 └─[目标] ida_artifact_gate → check_pointer_arithmetic.py（仅留 IDA 残留检查，B.5）
build_signals               → build_signals.py  生成 signals.json       （前置，非门）
audit_consistency.py        → 符号命名同步工具（B.4，非门）
pre_translate.py            → 翻译时工具（子Agent 翻译前自查偏移/被调可解析），非构建门；偏移正确性真正的保证来自 P2
```
- **B.2 与 B.5 是并行门、职责互补**，不是谁取代谁。
- **现状偏差**：B.5 目前**未接线**（之前被移除，原因是它**错误地用正则去做 B.2 的 Rule B**）。目标是把 B.5 重挂回，**只保留 IDA-残留标识符拦截**。
- 两道形式门 + 本行为门**都只查 git 改动函数**、**都吃同一个 `signals.json`**。

### B.1 `build_signals`（前置，非门）
`build_signals.py` 生成 `signals.json`（函数/类成员/全局；含 名字→偏移、是否指针、是否数组）。每次构建重生成，所以符号命名变化能及时反映（见 B.4）。

### B.2 `check_translated_functions.py`（形式/语法门，现役）
**作用域**：只查 `git diff HEAD` 改动函数。四阶段：
1. 取改动行：`git diff HEAD --unified=0 -- src/ app/`。
2. 花括号配平切出函数，只留与改动行有交集的。
3. 逐函数查 5 类违规（命中即 FAIL）：
   - **桩/注释壳（3 规则）**：真实代码 <3 → 桩；有纯注释行且真实代码 <5 → 注释壳；注释行占比 >50% → 注释壳。
   - **声明解析**：token 扫，把局部/形参分为指针/数组变量（区分函数级/块级作用域）。
   - **Rule A 指针算术**：`+`/`+=` 操作数是指针（先查 `signals.is_pointer`，回退本地声明），或 `(T*)expr+N`，或 `((T)ptr)(args)` → 违规；`->`/`.` 成员访问跳过。
   - **Rule B 非数组 `operator[]`**：`[` 前名字查 `signals.json`：`is_pointer==True` 违规、`is_array==False` 违规、`None` 回退本地声明。`[]` 只许用在确认是数组或本地 `T a[N]` 的名字上。
   - **C++ 转型 / goto**：`static_cast`/`dynamic_cast`/`reinterpret_cast`、`goto` → 违规。
4. 可选 `--ida-verify`（CMake 默认未开）：真实行比例对 IDA 指标判 LIKELY_STUB/BORDERLINE。
   退出码：任一违规 → 1。

### B.3 `verify_execution_flow` —— 本门（现役）
算法见 D。`--auto` **只读本地 IDA 缓存 `.omo/ida_cache/`、从不连 IDA MCP**；改动函数无缓存 → 跳过+WARNING（不挡构建）；**与 HEAD 流等价（含纯改名）→ 跳过（D-STEP4.5）**；**仅对 `completed=true` 函数施加严格裁决（D-STEP4.6）**；只有"有 IDA 数据、completed、严格核心对不上"才 exit 1（**FAIL 显示函数名、不显示函数内部信息**）。

### B.4 `audit_consistency.py` —— 符号命名同步工具（**非门**）
**很多调用数少的符号（`sub_*`/`dword_*`）尚未命名；设计上由子Agent 翻译时逐步补全 → 符号命名一定会随工作推进而变化。** 本工具职责是把这种变化在 IDA ↔ `functions.json` ↔ hpp 三处同步一致，**不挡构建**（当前 `--check-only` 审计，目标是同步）。这正是 C 节"按偏移不按名字"的根由——名字不稳定，偏移由二进制固定。

### B.5 `check_pointer_arithmetic.py`（**并行门，需重新接线，只保留 IDA-残留检查**）
保留并启用其独有能力：拦截 IDA 残留标识符 `vN`/`field_N`/`dword_N`/`sub_N`，以及 `(T*)0xNNNN`、`typeData[2048/4]`。**去掉**那段用正则重复 B.2 Rule-B 的逻辑。接回后"IDA 残留无人拦截"缺口闭合。

## C. 名词定义（仅项目/算法特定的；通用常识不赘述）

- **偏移 (offset)**：类成员从对象起始处算的**字节位置**（`BuildingClass::HasPower` 在第 1632 字节）。**按偏移比对、不按名字**——名字未完成且会随翻译演进（B.4），偏移由二进制固定。改名不改偏移 → 改名不改裁决（见 D-STEP4.5）。**偏移正确性（= 地址映射正确）是本门可信的硬前提**：用户强调"地址映射一定要确保正确"，后续翻译也依赖它；它完全正确时不存在无法解析的偏移（P2）。
- **基本块 (basic block)**：一段顺序执行、**中间不发生任何执行流分支**的连续语句；**凡导致执行流分支处都切分**（条件、循环、switch/case、跳转表、goto/label、return、可能改控制流的调用、异常等，任一处即切）。
- **边 / CFG**：基本块间的跳转；基本块 + 边 = 函数所有执行路径的控制流图。
- **事件 (event)**：①CALL ②WRITE(成员/全局) ③READ(成员/全局) ④RET；局部临时计算不算；按 偏移/地址 编码（`WRITE(member,1632)`），不按名字。**①②是"对外操作"= 严格核心；③READ = advisory（不裁决）；④RET 按存在性。** CALL 编码：具名调用→`CALL(0x解析地址)`、虚调用→`CALL(vfptr+槽偏移)`、解析不到的目标不产生事件（两侧同规则）。
- **后继集合**：一个基本块的后继是**一个基本块的集合**，按边的种类（真/假/各 case/默认/循环回边/直落）区分。
- **双射 / "保结构+保事件集合"**：IDA 侧每块与 C++ 侧某块配成唯一一对；保结构=两侧块及后继集合在配对下一一对应、边种类一致；保事件集合=每对块内**对外操作集合**一致。

## D. 核心算法（执行流怎么比）—— 计划主体

输入：原版函数的 IDA 伪代码（基准）+ 手写 C++ 翻译。判 C++ 能否被**证伪为"宏观结构/对外操作明显缺失"**。

**STEP 1 — 两侧各自建 CFG（基本块 + 边 + 每块"事件集合"）**
- 选 **IDA Hex-Rays 伪代码 CFG**（不是汇编 CFG）：伪代码已被 Hex-Rays 结构化（多数循环/switch 已还原回类源码形态），最大程度抹平编译器 CFG 差异。
- 事件**绑定到所属基本块**（不拍扁成函数级全局集合）——块绑定正是"把对外操作钉在结构骨架上"，能抓"某个块里的调用被注释掉了"。
- IDA 侧：`*(this+1632)` → `READ(member,1632)`；C++ 侧：`this->HasPower` →查 `signals.json` 得 1632 → `READ(member,1632)`。

**STEP 2 — 归一化（两侧同一套规则，消除"同一逻辑被写成不同形状"）**
- 短路 `||`/`&&` 拆分、早返回分支方向规整、直落合并、空块消除。**只动形状，不改"哪个块里有哪些对外操作"。**
- 现实控制流形态须建模到位（避免误报）：if/else、while/for/do、`?:`、短路、嵌套分支、**switch/case=跳转表=多路分支节点**（N case 后继+default）、goto/label、调用（含内联展开）、try/catch。
- **switch↔if-chain 形不同：靠作者契约（A 节）解决，不做强制归一化**——作者照伪代码形态写（伪代码是 if 树就写 if 树），所以"Hex-Rays 把稀疏 switch 降成 if 树、作者却写 switch"不会出现。循环倒置/尾复制 return 等：Hex-Rays 多已还原；P1 零误报样本若实测出残留误报，再在此补对应归一化。

**STEP 3 — 先做函数级必要条件检查，再做块绑定双射**

**(0) 函数级对外操作必要条件（廉价、与双射无关、永远运行——即使双射超时/被跳过）**：把 IDA 侧出现的每个 CALL 目标地址、每个 WRITE 偏移收成一个集合（先对**已翻译**的被内联被调做内联展开、并排除元素级豁免项），**这些必须在 C++ 侧至少各出现一次**；缺任一 → **FAIL**。这是个零误报的必要条件（忠实 C++ 必含原版每个对外操作；某 IDA CALL 目标在 C++ 里压根不出现 = 真缺失），**不依赖结构双射能否完成**，所以双射超时也兜得住"某对外操作被整个注释掉"。注：它只查"是否完全消失"、不查次数；次数级差异由 (1) 双射在不超时时抓，超时下的次数级差异是设计内允许的漏报（诚实声明）。**两侧对称**：C++ 侧"是否出现该 CALL/WRITE"同样是逐行文本扫描、不依赖 C++ CFG 建模——所以 C++ 侧 CFG 构建失败也不会把"C++ 里有没有"塌缩成假通过。

**(1) 块绑定双射**：入口块锚定，搜两侧块的一一配对，每对块满足：
- **CALL 集合一致、WRITE 集合一致**（对外操作 = 严格核心；按集合比，**块内先后次序不约束**——相对最早"敲死每块内顺序"版本的放宽）；
- **后继集合在配对下一一对应、边种类一致**（结构 = 严格核心）；
- **READ 不参与配对、不致 FAIL（advisory）**——Hex-Rays 与手写 C++ 的读连"集合有无"都会差（隐式 vtable 加载、辅助字段读、表达式因子分解），强比必误报；READ 至多作为多个候选双射并列时的次级排序提示，绝不单独否决配对。
- **内联**：原版内联了某小函数（IDA 侧有其体、无 CALL；C++ 侧一次 CALL）→ 把 C++ 侧被调函数 CFG 展开到调用点再比（现码 `efv_inline.try_inline_match`）。**被调未翻译/不可读时的处置（元素级、零误报且不漏逃逸）**：
  - 先看是否有"确信缺失的 IDA 侧 CALL/WRITE"——有就直接 **FAIL**（优先于 skip）；这正抓住"注释掉 CALL_X + 顺手调一个未翻译函数→想强制 skip"的逃逸。
  - 仅当未匹配的 IDA 侧内联区**不含任何 CALL/WRITE（只有 READ/RET，即平凡 getter 类内联）**、且这是唯一残留差异时 → **skip 该处 + WARN**（C++ 忠实调用了一个还没翻译的平凡 getter，挡它才是误报）。
  - 非平凡内联（区内有 CALL/WRITE）且被调未翻译 → 按作者契约应**镜像其内联体**或**先翻译被调**，否则 FAIL（不是误报：契约要求照伪代码形态写）。
  - 现码缺陷：此处现在一律保持 FAIL（平凡 getter 情形 = 误报）；P1 按上述元素级规则改。
- **双射超时/歧义**（大型 switch 重的函数、多个并列合法配对）→ **只跳过"结构/双射"裁决**（skip+WARN）；**但 STEP3(0) 的函数级对外操作必要条件检查照常运行、照常可 FAIL**——所以 timeout 也绝不漏掉"某对外操作在 C++ 完全消失"（堵住"大函数超时→整函数 skip→注释掉的 CALL 漏判"）。结构裁决本身绝不因超时变 FAIL（那才会误报）。

> 设计演进：v0=每块内对外操作**顺序敲死** → v1=现实现（CALL/WRITE/RET 块内按顺序、READ 集合，见现码 docstring 48–50 行）→ **v2(本计划目标)=CALL/WRITE 块内按集合、READ 降为 advisory、结构是唯一保留的顺序/形状不变量**。P1 第 1 项即把 v1 落到 v2。块绑定保留"对外操作钉在结构上"；因比对的是**伪代码**且 C++ 照伪代码形态转写（作者契约），块绑定在实践中稳定。

**STEP 4 — 二态裁决（严格核心 strict、其余宽松；无第三态）**
裁决只有 PASS / FAIL。**总原则（防"强制 skip"逃逸）**：**所有 skip 都是元素/区域级、绝不"整函数 skip 掩盖一个确信缺失的对外操作"。一旦（STEP3(0) 的函数级必要条件检查 或 块绑定双射）确认某 IDA 侧 CALL/WRITE 在 C++ 没有对应（= 被注释掉的对外操作特征）→ FAIL，且 FAIL 优先于一切 skip 路径——包括双射超时/歧义这种"结构裁决被跳过"的情形（必要条件检查仍跑）。** skip 只用来"豁免那个本身无法判定的元素"，绝不顺带豁免旁边一个独立确认缺失的 CALL/WRITE。
> **统一不变式（杜绝"全函数 skip 庇护所"）**：**唯一豁免 STEP3(0) 的整函数 skip 是"无 IDA 缓存"**（根本没有可抽取的 IDA 事件、无从比对）。**其余一切 skip——流等价 / completed=false / 未解析偏移 / 未翻译内联 / 双射超时歧义 / 未建模控制流形态——STEP3(0) 函数级对外操作必要条件检查都照常运行、照常可 FAIL**。因为对外操作的事件抽取是**逐行文本扫描、不依赖 CFG 建模/双射**，故凡有 IDA 伪代码可读，必要条件检查就能跑。
1. 配出"保结构+保对外操作集合"的完整双射 → **PASS**（仅表示"摸不出宏观差别"，非等价证明）。
2. **严格核心对不上**——C++ 侧相对原版**结构性缺块/缺分支，或某对应块缺 CALL/WRITE**（= 局部 stub 把对外操作注释掉的特征）→ **FAIL**。
   **反作弊与提示粒度**：FAIL **显示是哪个函数 +「未完成翻译」**（如 `Foo::Bar: 未完成翻译——可能存在局部 stub / 对外操作或结构缺失`），便于定位修复；但**不显示函数内部信息**（哪个事件/偏移/块差异），逼其补全整段而非靠内部线索贴补凑过。因比对本就粗（集合+结构），本来也无细粒度差异可泄露。其他门（B.2/B.5 等）的错误**照常显示**，互不影响。
3. C++ 用了本可避免、验证器读不了的写法（lambda/内联汇编、或可结构化掉的 goto）→ **FAIL + 提示换写法**（确定性写法前置，作者总能改，不算误报）。**例外**：原版伪代码本身是不可归约控制流、`goto` 无结构化等价（罕见）→ 这类**只跳过结构裁决（skip + WARN），STEP3(0) 必要条件检查照常运行、照常可 FAIL**（同未建模形态，绝不整函数放过）；其是否允许 goto 是 B.2 的项目策略，另议。
4. **偏移解析以二进制为准、是 P2 硬前提**：用户强调"地址映射一定要确保正确"——地址映射完全正确时**根本不存在未解析偏移**（符号转写定义此前应已对应好，P2 主要是核对 + 修已知错如 ObjectTypeClass）。**过渡期**若仍遇未解析偏移 → **元素级 skip：只豁免那个无法编码的成员 WRITE/READ + WARNING + 记入"偏移待补/P2"清单（绝不 FAIL）**。但**CALL 按地址独立解析、不受成员偏移影响**——所以即便同块有未解析成员，旁边"确信缺失的 CALL/WRITE"仍 FAIL（不被未解析偏移顺带豁免，堵"注释掉 CALL + 写一个未映射成员→想强制 skip"的逃逸）。
   > 现码必修缺陷：现在 C++ 侧未解析成员发 `(member,?name)` 哨兵"永不匹配"→ 双射失败 → **FAIL = 误报**。P1 改成"该元素豁免、不致 FAIL；其余照判"。
   > 取消上版"偏移 fail-closed / 归 pre_translate"措辞——pre_translate 是翻译时工具、非构建门；可信保证来自 P2 地址映射正确，不靠构建期接线 pre_translate。
5. **流等价跳过（纯改名/无对外行为变化的改动不门控）**：批量重命名让 `git diff` 标记大片改动。判定"是否跳过"**必须按偏移流等价、绝不用文本启发式**——把该函数当前的（CFG + CALL/WRITE 集合）与其 **HEAD 版本**比，**完全一致才 skip**。
   - 安全性：若混入"注释掉一个 CALL"，对外操作集合就变了 → 不流等价 → **照常门控**（堵住"改名 + 注释掉调用"的混合 diff 逃逸；纯文本判定会漏掉这种逃逸，故禁用）。
   - 代码现状（已核对 `get_modified_functions`）：它只挑**带函数体 + IDA 地址注释的 cpp 定义**，hpp 纯声明改名不会被纳入；定义/实现已分离，改名只触及 hpp 与"使用该符号的 cpp 体"，被触及的**已完成**函数偏移不变 → PASS（无 churn）。
   - 隐含假设（HEAD 已过门）：流等价跳过只在"与 HEAD 完全一致"时成立，依赖 HEAD 当初已过门（或因无缓存/completed=false 未门控）。被改名触及的函数体会重新进入 B.2 改动集、且下游对拍兜底，故此优化安全、非裸断言。
6. **completed 作用域（解决"改名误挡在建 stub"）**：本门**只对 `functions.json` 中 `completed=true`（已声明 RE 实现完整）的函数施加严格裁决**。被改名顺带扫到、但仍 `completed=false` 的**在建/stub 函数不被本门纳入**（其半成品由 B.2 桩检测负责），不会因一次无关改名被本门误挡。`completed=true` 后再注释掉 CALL → 仍被本门抓住。现码已加载 `name_to_completed`（来自 `hook.completed`），P1 接上此门控即可。

**未建模的新控制流形态（罕见）**：属**验证器自身覆盖缺陷 = 工具债**。**只跳过"结构/双射"裁决（WARNING + 记入"覆盖待补"清单），STEP3(0) 函数级对外操作必要条件检查照常运行、照常可 FAIL**——对外操作事件抽取是逐行文本扫描、不依赖 CFG 建模，未建模形态下仍能抽出 CALL 目标/WRITE 偏移集合。**这与"无 IDA 缓存"不同**（后者根本无可抽取事件、才整函数跳过）——不可把未建模形态当无缓存那样整函数放过，否则就成了与超时同构的庇护所。目标是把现实会出现的形态全建模、让清单清空。（正常函数的"结构+对外操作"核心始终严格阻断；未建模/超时函数的"对外操作必要条件"也始终阻断。）

## H. 验收（怎么算"有效工作"）
- **零误报（首要、硬指标）**：忠实样本（足量、含编译器变换花样 + 跨块代码移动 + **未建模/不可归约等最少测过的伪代码形状**）上 **0 误挡**；给出样本量与置信说明（非"十来个"）。未解析偏移 → skip（不再误报）。
- **崩溃路径隔离（实现义务，最高优先）**：IDA 侧 CFG 构建 / C++ 侧解析或 CFG 构建 / 归一化里的**未捕获异常，绝不可静默整函数放过**——必须或 FAIL、或退到"STEP3(0) 仍运行"。用**故障注入测试**证明：强制 STEP1 在一个被删了 resolvable CALL 的函数上抛异常 → 仍必须 FAIL（被 STEP3(0) 抓）。这是把"STEP3(0) 永远运行"从断言变成可验证的工程保证。
- **粗召回成立**：截断/删分支/删 CALL 簇/删 WRITE + 历史真实逃逸样本**全部 FAIL**。
- **skip 不掩盖逃逸（关键）**：把"注释掉一个对外操作"分别与**每个 skip 触发器**（未翻译内联被调 / 未解析偏移 / 双射超时歧义 / 未建模控制流形态）组合，**仍必须 FAIL**——前两者靠元素级 skip + 缺失优先 FAIL，**超时歧义 / 未建模形态靠 STEP3(0) 函数级必要条件检查兜底**（与双射/CFG 建模无关、凡有伪代码可读就运行）。验证 skip 绝不顺带豁免独立确认缺失的对外操作。
- **skip 率预算/监控**：统计生产函数里各 skip 路径（无缓存/流等价/completed=false/未解析偏移/未翻译内联/超时歧义/未建模）的命中率并纳入报告——零误报但大面积 skip 的门没有保证；skip 率过高即覆盖告警。**单列"超时/歧义"与"未建模形态"两个桶**——它们是仅靠函数级必要条件兜底的路径，其 skip 率上升是 mandate 泄漏的早警。
- **流等价跳过安全**：纯改名/与 HEAD 流等价的改动跳过；"改名 + 注释掉 CALL"的混合 diff **不被误判为改名**、照常门控。
- **completed 作用域成立**：`completed=false` 的在建/stub 不被本门误挡；`completed=true` 后注释掉 CALL 仍被抓。
- **地址映射正确**：偏移经二进制核对、ObjectTypeClass=660 等已修；不存在偏移错位导致的误报。
- **设计边界诚实**：②条件/运算符/常量、③对象身份、变量下标 等 known-limitation **明确文档化为"设计内允许的漏报"**，不被当成 bug、也不靠裁剪变异样本掩盖；并注明**下游对拍本身也受输入覆盖限制**（概率性兜底，非绝对保证）。
- **严格核心确实严格**：结构（块/后继/边种类）+ 对外操作（CALL/WRITE 块绑定集合）任一确信缺失 → FAIL。
- **反作弊成立**：FAIL 显示函数名 + "未完成翻译"，**不显示函数内部信息**；人工 verbose 模式不进子Agent 回路；他门错误照常显示。
- **覆盖面成立**：switch/跳转表按多路分支建模、switch↔if 由作者契约覆盖；罕见未建模形态走跳过+WARNING+待补清单（仅限工具债，非正常函数放水）。
